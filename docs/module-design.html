▶
C0.1EnvConfigLoader (AppProperties) ▶
application.yml의 커스텀 설정을 타입 안전하게 바인딩하는 설정 클래스
IN (메인)
application.yml, application-{profile}.yml — 파일 시스템에서 로드되는 설정 파일
OUT
AppProperties {
  jwtSecret: String — JWT 서명 비밀 키
  jwtAccessExpiry: long — 액세스 토큰 만료 시간 (ms)
  jwtRefreshExpiry: long — 리프레시 토큰 만료 시간 (ms)
  googleClientId: String — Google OAuth2 클라이언트 ID
  allowedOrigins: List<String> — CORS 허용 도메인 목록
}
내부 로직
  1. @ConfigurationProperties(prefix = "app")로 "app." 접두사 설정을 바인딩
  2. @Validated로 필수값 누락 시 애플리케이션 시작 실패 처리
  3. 프로파일(dev/prod)별 설정 오버라이드 적용
제약사항
jwtSecret은 최소 256비트(32자) 이상 필수. 프로덕션 환경에서는 반드시 환경변수로 주입.
C0.2DatabaseConfig ▶
MySQL DataSource + JPA/Hibernate 설정
IN (보조)
C0.1의 DB URL, username, password — application.yml의 spring.datasource.* 설정
OUT
DataSource — HikariCP 기반 커넥션 풀
EntityManagerFactory — JPA 엔티티 매니저 팩토리
PlatformTransactionManager — 트랜잭션 매니저
내부 로직
  1. HikariCP 커넥션 풀 생성 (최대 연결 수: 10개)
  2. Hibernate ddl-auto=validate 설정 (스키마 검증만, 변경은 Flyway 담당)
  3. open-in-view=false 설정 (지연 로딩 문제 방지)
  4. MySQL 8.4 방언(dialect) 자동 감지
제약사항
ddl-auto는 반드시 validate. 스키마 변경은 Flyway 마이그레이션 파일로만 수행.
C0.3SecurityConfig ▶
Spring Security 필터 체인 설정 (CORS, CSRF, JWT 필터 등록)
IN (보조)
C0.5 JwtAuthFilter — Spring Bean으로 주입되는 JWT 인증 필터
OUT
SecurityFilterChain — Spring Security 필터 체인 빈
내부 로직
  1. CSRF 비활성화 (stateless REST API이므로 불필요)
  2. 세션 관리 정책: STATELESS (JWT 기반 인증)
  3. 허용 경로 설정: /api/auth/** — 인증 없이 접근 가능
  4. 나머지 모든 경로: 인증 필수 (authenticated())
  5. JwtAuthFilter를 UsernamePasswordAuthenticationFilter 앞에 등록
  6. CORS 설정: allowedOrigins에서 Flutter 웹앱 도메인 허용
제약사항
프로덕션에서 allowedOrigins를 정확히 지정. 와일드카드(*) 사용 금지.
C0.4JwtTokenProvider ▶
JWT 액세스/리프레시 토큰 생성 및 검증
IN (메인)
userId: Long — 사용자 고유 ID
email: String — 사용자 이메일
IN (보조)
C0.1의 jwtSecret: String, jwtAccessExpiry: long, jwtRefreshExpiry: long
OUT
TokenPair {
  accessToken: String — 30분 유효 액세스 토큰
  refreshToken: String — 7일 유효 리프레시 토큰
}

메서드:
  generateTokenPair(userId, email) → TokenPair
  validateToken(token) → boolean
  getUserIdFromToken(token) → Long
  getEmailFromToken(token) → String
내부 로직
  1. HMAC-SHA256 알고리즘으로 토큰 서명
  2. Claims에 userId + email 포함
  3. accessToken 만료: 30분, refreshToken 만료: 7일
  4. validateToken — 서명 검증 + 만료 확인
제약사항
토큰 만료 시 validateToken이 false 반환. 별도 예외 미발생.
C0.5JwtAuthFilter ▶
HTTP 요청의 Authorization 헤더에서 JWT를 추출하여 SecurityContext 설정
IN (메인)
HttpServletRequest — Authorization: Bearer {token} 헤더 포함
IN (보조)
C0.4 JwtTokenProvider — 토큰 검증 및 정보 추출 도구
OUT
SecurityContext에 Authentication 객체 설정 — userId: Long, authorities: Collection
내부 로직
  1. Authorization 헤더에서 "Bearer " 접두사 제거 → 토큰 추출
  2. C0.4.validateToken(token) 호출하여 유효성 확인
  3. 유효하면 userId, email 추출 → UsernamePasswordAuthenticationToken 생성
  4. SecurityContextHolder에 인증 객체 설정
  5. 토큰이 없거나 유효하지 않으면 → 필터 통과 (SecurityConfig에서 접근 거부 처리)
제약사항
OncePerRequestFilter 상속. /api/auth/** 경로는 필터 스킵.
C0.6OAuth2UserLoader ▶
Google OAuth2 ID 토큰을 검증하고 사용자 정보 추출
IN (메인)
idToken: String — Google에서 발급한 ID 토큰
IN (보조)
C0.1의 googleClientId: String
OUT
GoogleUserInfo {
  googleId: String — Google 고유 사용자 ID (sub 클레임)
  email: String — 사용자 이메일
  name: String — 사용자 표시 이름
  pictureUrl: String — 프로필 사진 URL
}
내부 로직
  1. Google의 tokeninfo 엔드포인트 또는 google-auth-library로 ID 토큰 검증
  2. aud(audience)가 googleClientId와 일치하는지 확인
  3. 토큰에서 sub(googleId), email, name, picture 추출
제약사항
만료된 토큰이나 aud 불일치 시 ApiException(INVALID_TOKEN) 발생.
C0.7ApiException + ErrorCode ▶
비즈니스 에러를 표현하는 커스텀 예외 및 에러코드 정의
IN (메인)
errorCode: ErrorCode — 에러코드 enum 값
message: String (선택) — 커스텀 에러 메시지
OUT
ApiException {
  errorCode: ErrorCode — 에러코드
  message: String — 에러 메시지
  httpStatus: HttpStatus — HTTP 상태 코드
}

ErrorCode enum:
  AUTH_FAILED(401, "인증에 실패했습니다")
  TOKEN_EXPIRED(401, "토큰이 만료되었습니다")
  INVALID_TOKEN(401, "유효하지 않은 토큰입니다")
  FORBIDDEN(403, "접근 권한이 없습니다")
  NOT_FOUND(404, "리소스를 찾을 수 없습니다")
  DUPLICATE(409, "이미 존재하는 리소스입니다")
  VALIDATION_FAILED(422, "입력값이 유효하지 않습니다")
  INTERNAL_ERROR(500, "서버 내부 오류가 발생했습니다")
  TIME_LOCKED(422, "시간 잠금: 과거 날짜의 습관은 수정할 수 없습니다")
내부 로직
  1. RuntimeException을 상속하여 비검사 예외로 정의
  2. ErrorCode enum에 HTTP 상태 코드와 기본 메시지를 정의
  3. 커스텀 메시지가 없으면 ErrorCode의 기본 메시지 사용
C0.8GlobalExceptionHandler ▶
@RestControllerAdvice로 전역 예외를 ApiResponse 형태로 변환
IN (메인)
Exception — 컨트롤러에서 발생한 예외
  • ApiException — 비즈니스 예외
  • MethodArgumentNotValidException — @Valid 검증 실패
  • Exception — 기타 예외
OUT
ResponseEntity<ApiResponse<Void>> — 에러 응답 래퍼
내부 로직
  1. ApiException → errorCode의 httpStatus + message로 응답 생성
  2. MethodArgumentNotValidException → 422 + 필드별 에러 메시지 목록 반환
  3. 기타 Exception → 500 + "서버 내부 오류" 응답 생성
제약사항
스택 트레이스는 로그에만 출력. 클라이언트 응답에 스택 트레이스 포함 금지.
C0.9ApiResponse<T> ▶
모든 API 응답의 통일된 래퍼 클래스
IN (메인)
success: boolean — 성공 여부
data: T (nullable) — 응답 데이터
message: String (nullable) — 메시지
OUT
ApiResponse<T> {
  success: boolean — 요청 성공 여부
  data: T — 응답 데이터 (실패 시 null)
  message: String — 메시지 (에러 시 에러 메시지)
  timestamp: LocalDateTime — 응답 생성 시각
}

정적 팩토리 메서드:
  ApiResponse.success(data) → { success: true, data, message: null }
  ApiResponse.success(data, message) → { success: true, data, message }
  ApiResponse.error(message) → { success: false, data: null, message }
제약사항
모든 Controller는 반드시 ApiResponse<T>로 래핑하여 반환해야 함.

CREATE TABLE users (

CREATE TABLE events (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    title VARCHAR(200) NOT NULL,
    description TEXT,
    event_type ENUM('NORMAL','RANGE','RECURRING','TODO') NOT NULL DEFAULT 'NORMAL',
    start_date DATETIME NOT NULL,
    end_date DATETIME,
    all_day BOOLEAN NOT NULL DEFAULT FALSE,
    color VARCHAR(7) DEFAULT '#5B8DEF',
    location VARCHAR(300),
    recurrence_rule VARCHAR(200),
    range_tag ENUM('TRAVEL','EXAM','VACATION','PROJECT','OTHER'),
    memo TEXT,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    INDEX idx_events_user_date (user_id, start_date),
    INDEX idx_events_user_type (user_id, event_type)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
