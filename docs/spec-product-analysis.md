# Design Your Life - 제품/사용자 관점 분석서 (spec-product)

---

## 1. 사용자 시나리오 (User Scenarios)

### 1.1 플랫폼별 사용 맥락 차이

| 구분 | 웹 (Flutter Web) | 앱 (Android) |
|---|---|---|
| 주요 사용 맥락 | 데스크톱/노트북 앞에서 집중 계획 수립 | 이동 중, 실시간 체크, 빠른 기록 |
| 세션 평균 시간 | 5~15분 (긴 세션) | 30초~3분 (짧은 세션) |
| 핵심 행동 | 목표 설정, 만다라트 작성, 주간 루틴 편집, 통계 리뷰 | 습관 체크, 투두 완료, 일정 확인 |
| 입력 방식 | 키보드 + 마우스 (대량 입력 유리) | 터치 (탭 위주, 입력 최소화) |
| 화면 크기 | 넓은 화면 활용 가능 (사이드바, 멀티컬럼) | 세로 스크롤 중심, 단일 컬럼 |

### 1.2 시나리오 A: 최초 진입 사용자 (온보딩)

**페르소나**: 김서연, 25세, 대학생. 토익 준비와 운동 습관을 동시에 관리하고 싶다.

**웹 시나리오**:
1. Google 검색 또는 SNS 링크로 웹앱에 접근한다.
2. 랜딩 화면에서 "Google로 시작하기" 버튼을 탭한다.
3. Google OAuth 동의 화면이 표시되고, 계정을 선택한다.
4. 이름 입력 화면이 표시된다 ("어떻게 불러드릴까요?").
5. 이름을 입력하면 홈 대시보드로 이동한다.
6. 대시보드에 빈 상태 카드들이 표시되고, 각 카드에 "첫 OOO을 추가해보세요!" 안내가 있다.

**앱 시나리오**:
1. Google Play Store에서 "Design Your Life"를 설치한다.
2. 앱을 열면 스플래시 화면 후 로그인 화면이 표시된다.
3. "Google로 시작하기" 버튼을 탭한다.
4. 기기에 등록된 Google 계정 목록에서 선택한다.
5. 이름 입력 화면 ("어떻게 불러드릴까요?") 후 홈으로 이동한다.
6. 웹에서 이미 데이터가 있다면 동기화되어 바로 표시된다.

**핵심 설계 원칙**: 로그인부터 첫 화면까지 3탭 이내. 별도 튜토리얼 화면 없이 빈 상태 UI가 곧 가이드 역할을 한다.

### 1.3 시나리오 B: 일상 루틴 사용자 (Daily Use)

**페르소나**: 박지훈, 28세, 직장인. 매일 아침 앱을 열어 오늘 할 일을 확인한다.

**아침 루틴 (앱, 07:00)**:
1. 앱을 연다 -> 홈 대시보드가 표시된다.
2. "안녕하세요, 지훈님!" 인사 메시지와 오늘 날짜를 확인한다.
3. 오늘의 투두 완료율 도넛차트 (0%)와 할 일 목록 미리보기를 본다.
4. 오늘의 습관 달성률 (0%)과 습관 카드 미리보기를 본다.
5. 다가오는 D-day 카드에서 "토익 시험 D-7" 빨간색 카드를 확인한다.
6. 투두 탭으로 이동하여 할 일 목록을 체크하기 시작한다.

**퇴근 후 (앱, 19:00)**:
1. 홈 대시보드에서 투두 완료율 65%를 확인한다.
2. 습관 탭으로 이동하여 "운동 30분", "물 2L" 등을 체크한다.
3. 스트릭 카운터가 "12일 연속"으로 증가하는 것을 확인한다.
4. 홈으로 돌아오면 습관 달성률이 업데이트되어 있다.

**주말 계획 (웹, 일요일 저녁)**:
1. 웹에서 캘린더 탭의 주간 뷰를 연다.
2. 다음 주 일정을 확인하고 새 일정을 추가한다.
3. 목표 탭에서 월간 목표 진행률을 리뷰한다.
4. 루틴 탭에서 다음 주 시간표를 조정한다.

### 1.4 시나리오 C: 캘린더 일정 생성 (4가지 유형)

**일반 일정 생성**:
1. 캘린더 탭에서 특정 날짜를 탭한다.
2. 하단에 해당 날짜 일정 리스트가 표시된다.
3. "+" 버튼을 탭하면 일정 유형 선택 바텀시트가 올라온다.
4. "일반 일정"을 선택한다.
5. 제목, 시간, 색상(8가지), 위치, 메모를 입력한다.
6. "저장"을 탭하면 캘린더에 dot이 표시된다.

**범위 일정 생성**:
1. "+" 버튼 -> "범위 일정" 선택.
2. 제목을 입력한다 (placeholder: "예: 제주도 여행").
3. 일정 유형 태그를 선택한다 (여행/시험/휴가/프로젝트/기타).
4. 시작일과 종료일을 선택한다 (기간이 자동 계산되어 "N일간" 표시).
5. 색상, 위치, 메모를 입력한다.
6. 저장하면 캘린더에서 범위가 시각적으로 연결되어 표시된다.

**반복 일정 생성**:
1. "+" 버튼 -> "반복 일정" 선택.
2. 일정 이름을 입력한다 (placeholder: "예: 영어 수업").
3. 반복 요일을 선택한다 (월~일 중 복수 선택 가능).
4. 시작 시간과 종료 시간을 설정한다.
5. 색상을 선택한다.
6. 저장하면 선택된 요일마다 캘린더에 자동 반영된다.

**할일 일정 생성**:
1. "+" 버튼 -> "할일" 선택.
2. 할 일 제목, 시간 지정 토글, 색상, 메모를 입력한다.
3. 시간 지정을 켜면 특정 시간에 타임라인에 표시된다.
4. 시간 지정을 끄면 해당 날짜의 할 일 목록에만 표시된다.

### 1.5 시나리오 D: 습관 트래킹 (시간 잠금 메커니즘)

**첫 습관 등록**:
1. 습관 탭 -> "오늘의 습관" 섹션이 빈 상태로 표시된다.
2. "인기 습관으로 시작하기" 버튼이 표시된다.
3. 프리셋 목록 (운동/독서/물마시기/영어/일기)에서 원하는 것을 선택한다.
4. 선택 즉시 습관 카드가 생성되고 체크 가능 상태가 된다.

**일일 체크 흐름**:
1. 습관 카드에서 체크박스를 탭한다.
2. 체크 시 스트릭 카운터가 +1 증가한다.
3. 도넛차트가 실시간 업데이트된다 (예: 2/5 -> 3/5, 40% -> 60%).
4. 모든 습관 완료 시 축하 피드백이 표시된다.

**시간 잠금 규칙**:
- 오늘(00:00~23:59)의 습관만 체크/해제 가능하다.
- 자정이 지나면 어제 습관은 잠금 상태가 된다 (체크 불가, 회색 표시).
- 어제 체크하지 않은 습관의 스트릭은 0으로 리셋된다.
- 과거 날짜의 습관 캘린더는 읽기 전용이다.

### 1.6 시나리오 E: 목표 계층 관리

**년간 목표 설정 (웹 권장)**:
1. 목표 탭 -> "년간 목표" 서브탭을 선택한다.
2. "+" 버튼으로 새 목표를 추가한다.
3. 목표 이름 (예: "토익 800점"), 설명을 입력한다.
4. 저장하면 진행률 0% 카드가 생성된다.

**월간 하위목표 연결**:
1. 년간 목표 카드를 탭하여 상세 화면에 진입한다.
2. "월간 하위목표 추가" 버튼을 탭한다.
3. 하위목표 이름 (예: "3월 - LC 파트 집중")을 입력한다.
4. 해당 하위목표 아래에 실천 할일을 추가한다.
5. 실천 할일을 완료할 때마다 하위목표 진행률이 자동 계산된다.
6. 하위목표 진행률의 평균이 년간 목표 진행률에 반영된다.

**상단 통계 자동 갱신**:
- 달성률(%): 완료 표시된 목표 수 / 전체 목표 수.
- 평균 진행률(%): 모든 목표의 진행률 평균.
- 총 목표 수: 활성 목표 개수.

### 1.7 시나리오 F: 만다라트 생성 (단계별 위저드)

**1단계 - 핵심 목표 입력**:
1. 목표 탭 -> "만다라트" 서브탭으로 전환한다.
2. 빈 상태에서 "만다라트란?" 툴팁 아이콘이 있다.
3. 툴팁을 탭하면 만다라트 개념 설명이 표시된다.
4. "만다라트 만들기" 버튼을 탭한다.
5. "핵심 목표를 입력하세요" 화면에서 중심 목표를 입력한다 (예: "2026년 나의 성장").

**2단계 - 세부목표 8개 입력**:
1. 핵심 목표가 중앙에 표시되고, 주변 8칸이 빈 상태로 표시된다.
2. 각 칸을 탭하여 세부목표를 입력한다 (예: 건강, 재정, 커리어, 관계, 학습, 취미, 마인드, 환경).
3. 모두 입력하지 않아도 다음으로 진행 가능하다 (선택적).

**3단계 - 실천과제 입력**:
1. 각 세부목표를 탭하면 해당 세부목표 주변 8칸이 표시된다.
2. 실천과제를 하나씩 입력한다.
3. 실천과제는 할일과 연동되어 체크 시 진행률에 반영된다.

**완성 후**:
- 9x9 그리드 전체가 채워진 만다라트가 시각적으로 표시된다.
- 완료된 실천과제는 색상이 변경되어 진행 상황을 한눈에 파악 가능하다.
- 목표 리스트 뷰와 만다라트 뷰는 동일 데이터를 다른 시각으로 보여준다.

### 1.8 시나리오 G: 루틴 등록 (주간 시간표)

1. 습관 탭 -> "내 루틴" 서브탭으로 전환한다.
2. "+" 버튼으로 새 루틴을 추가한다.
3. 바텀시트에서 일정 이름 (placeholder: "예: 영어 수업"), 반복 요일 (월~일 다중 선택), 시작/종료 시간, 색상을 입력한다.
4. 저장하면 루틴 카드가 생성된다 (이름, 요일 뱃지, 시간, 활성/비활성 토글).
5. 활성 토글을 켜면 캘린더 탭의 주간/일간 타임라인에 해당 루틴이 자동 반영된다.
6. 비활성으로 전환하면 캘린더에서 숨겨진다 (루틴 데이터는 유지).

## 2. 진입장벽 완화 전략 상세화

### 2.1 습관: 인기 프리셋 전략

| 프리셋 이름 | 아이콘 | 설명 | 선택 이유 |
|---|---|---|---|
| 운동 30분 | 운동 아이콘 | 매일 30분 이상 운동하기 | 가장 보편적 자기계발 습관 |
| 독서 | 책 아이콘 | 매일 독서하기 | 생산성 앱 타겟층의 핵심 관심사 |
| 물 2L | 물방울 아이콘 | 하루 물 2리터 마시기 | 건강 습관의 가장 쉬운 시작점 |
| 영어 공부 | 말풍선 아이콘 | 매일 영어 학습하기 | 20~30대 주요 관심사 |
| 일기 쓰기 | 펜 아이콘 | 매일 일기 쓰기 | 자기성찰 습관, 감성 호소 |

**UX 흐름**:
- 첫 진입 시 "인기 습관으로 시작하기" 카드가 상단에 표시된다.
- 프리셋을 탭하면 바로 습관이 등록된다 (추가 입력 불필요).
- 프리셋 하단에 "직접 만들기" 옵션을 제공하여 커스텀 습관도 지원한다.
- 프리셋 선택 후에도 이름/아이콘 수정이 가능하다.

### 2.2 루틴: 빈 상태 가이드

**빈 상태 화면 구성**:
- 중앙에 시간표 일러스트 아이콘을 배치한다.
- "아직 등록된 루틴이 없어요" 메인 텍스트.
- "반복되는 일정을 루틴으로 등록하면, 캘린더에 자동으로 표시돼요!" 서브 텍스트.
- "첫 루틴 만들기" CTA 버튼.

**예시 가이드**: 루틴 추가 바텀시트의 이름 입력 필드에 placeholder로 "예: 영어 수업"을 제공하여 사용 맥락을 암시한다.

### 2.3 목표: 템플릿 전략

| 템플릿 이름 | 하위목표 예시 | 타겟 사용자 |
|---|---|---|
| 토익 800+ | LC 파트 집중, RC 문법 정리, 모의고사 풀기 | 취업 준비생 |
| 정보처리기사 | 필기 이론 정리, 기출 풀이, 실기 연습 | IT 취준생/직장인 |
| 다이어트 -5kg | 식단 관리, 운동 루틴, 체중 기록 | 건강 관심 사용자 |
| 영어 회화 | 매일 회화 연습, 영어 일기, 원서 읽기 | 자기계발 관심층 |

**UX 흐름**:
- 목표 리스트가 비어있을 때 "템플릿으로 시작하기" 옵션을 표시한다.
- 템플릿 선택 시 년간 목표 + 월간 하위목표가 자동 생성된다.
- 사용자는 생성된 구조를 자유롭게 수정/삭제할 수 있다.

### 2.4 만다라트: 개념 설명 + 단계별 위저드

**진입장벽 분석**: 만다라트는 일반 사용자에게 낯선 개념이다. 9x9 그리드를 한 번에 보여주면 압도감을 준다.

**완화 전략**:
1. "만다라트란?" 툴팁: 만다라트의 의미와 활용법을 간략히 설명한다. "핵심 목표 1개 -> 세부 목표 8개 -> 각각의 실천 과제 8개를 구조화하는 목표 관리 기법이에요."
2. 단계별 위저드: 한 번에 9x9를 채우지 않고 3단계로 분리한다.
   - 1단계: 핵심 목표 1개만 입력 (부담 최소화).
   - 2단계: 세부 목표 8개 입력 (건너뛰기 가능).
   - 3단계: 실천 과제 입력 (세부 목표 탭 시 펼침).
3. 부분 저장 허용: 모든 칸을 채우지 않아도 저장 가능하다. 빈 칸은 "탭하여 추가" 상태로 유지된다.
4. 목표 리스트와 연동: 만다라트에서 입력한 목표는 목표 리스트에도 동일하게 반영된다.

### 2.5 홈: 최소한의 온보딩

**설계 원칙**: 별도의 온보딩 튜토리얼 화면을 만들지 않는다. 온보딩 = 빈 상태 UI.

**흐름**:
1. Google 로그인 성공.
2. 이름 입력 (단일 입력 필드, "시작하기" 버튼).
3. 홈 대시보드 진입 -> 모든 섹션이 빈 상태이지만, 각 카드가 자체적으로 행동 유도 메시지를 포함한다.

**사용자 가치**: 10초 이내에 앱 사용을 시작할 수 있다. 기능을 쓰면서 자연스럽게 학습한다.

2. **탭 간 연결 최소 성립**: 5개 탭 모두 빈 껍데기가 아닌, 각각 최소 1개 핵심 기능이 동작하는가?
3. **데이터 흐름 완결**: 입력 -> 저장 -> 표시 -> 수정 -> 삭제가 모든 데이터에서 동작하는가?
4. **과도한 복잡성 배제**: 구현 복잡도 대비 사용자 가치가 낮은 기능은 후속 버전으로 미루는가?

## 5. 비즈니스 가치 평가

### 5.1 시장 포지셔닝

**현재 시장 문제**: 생산성 앱 시장은 파편화되어 있다.
- 투두 앱 (Todoist, TickTick): 할 일만 관리.
- 캘린더 앱 (Google Calendar, Apple Calendar): 일정만 관리.
- 습관 앱 (Habitica, Streaks): 습관만 관리.
- 목표 앱 (Notion 수동 구성): 설정이 복잡하다.

**Design Your Life의 포지션**: "4가지를 하나의 앱에서 관리하는 올인원 생산성 대시보드"

### 5.2 핵심 차별 가치

| 가치 | 설명 |
|---|---|
| 올인원 통합 | 투두+캘린더+습관+목표를 하나의 앱에서 관리. 앱 전환 비용 제거 |
| 데이터 연결 | 목표 -> 할일, 루틴 -> 캘린더, 습관 -> 대시보드. 데이터가 서로 연결되어 시너지 발생 |
| 시각적 동기부여 | 도넛차트, 진행률 바, 스트릭 카운터, 만다라트 그리드. 성취감을 시각화 |
| 낮은 진입장벽 | 프리셋, 템플릿, 빈 상태 가이드. 생산성 도구의 최대 약점인 "시작의 어려움" 해결 |
| 웹+앱 동기화 | 데스크톱에서 계획, 모바일에서 실행. 사용 맥락에 맞는 크로스플랫폼 |

### 5.3 타겟 사용자 세그먼트

| 세그먼트 | 특징 | 핵심 가치 | 예상 비중 |
|---|---|---|---|
| 취업 준비생 | 토익/자격증 목표, 일정 관리 필요 | 목표 계층 + 투두 + 캘린더 | 30% |
| 대학생 | 수업 시간표, 과제 관리, 습관 형성 | 루틴 + 투두 + 습관 | 25% |
| 주니어 직장인 | 업무/개인 일정 분리, 자기계발 습관 | 캘린더 + 습관 + 목표 | 25% |
| 자기계발 관심층 | 다양한 목표 동시 추구, 만다라트 관심 | 만다라트 + 목표 + 습관 | 20% |

### 5.4 리텐션 설계 포인트

1. **스트릭 심리**: 연속 기록이 쌓이면 깨기 싫은 심리가 작용한다. 습관 스트릭은 DAU 핵심 드라이버이다.
2. **일일 대시보드**: 매일 아침 "오늘의 상태"를 확인하는 습관을 형성한다. 0%에서 100%로 채워가는 성취감.
3. **D-day 긴급성**: 다가오는 일정의 D-day 카운트가 앱 진입 동기를 부여한다.
4. **진행률 시각화**: 목표의 진행률 바가 채워지는 것을 보는 만족감이 지속 사용을 유도한다.
5. **시간 잠금**: 오늘만 체크 가능한 제약이 "지금 해야 한다"는 행동 촉구를 만든다.

### 5.5 수익화 가능성 (v2.0 이후)

| 모델 | 내용 | 시점 |
|---|---|---|
| 프리미엄 구독 | 무제한 습관/목표, 통계 리포트, 테마 커스터마이징 | v2.0 이후 |
| 광고 (배너) | 무료 사용자 대상 비침입적 광고 | v1.5 이후 |
| 템플릿 마켓 | 사용자 제작 만다라트/목표 템플릿 공유/판매 | v3.0 이후 |

MVP 단계에서는 수익화를 하지 않는다. 사용자 기반 확보와 리텐션 검증이 우선이다.

## 6. 빈 상태(Empty State) UI 전략

### 6.1 설계 원칙

모든 빈 상태 UI는 3가지 요소를 포함한다:
1. **시각적 요소**: 해당 기능을 상징하는 아이콘 또는 일러스트.
2. **안내 텍스트**: "무엇이 없다" + "무엇을 하면 된다"를 한 쌍으로 제공.
3. **행동 유도 (CTA)**: 바로 데이터를 추가할 수 있는 버튼 또는 프리셋 제안.

### 6.2 탭별 빈 상태 상세

#### 홈 대시보드

| 섹션 | 빈 상태 텍스트 | CTA |
|---|---|---|
| 오늘의 투두 | "오늘 할 일이 없어요" / "투두 탭에서 할 일을 추가해보세요!" | "할 일 추가하러 가기" (투두 탭 이동) |
| 오늘의 습관 | "등록된 습관이 없어요" / "작은 습관부터 시작해보세요!" | "습관 등록하러 가기" (습관 탭 이동) |
| D-day | "다가오는 일정이 없어요" / "캘린더에서 일정을 추가해보세요!" | "일정 추가하러 가기" (캘린더 탭 이동) |
| 이번주 요약 | "이번 주 데이터가 아직 없어요" / "할 일과 습관을 시작하면 자동으로 요약돼요!" | 없음 (자동 생성 영역) |

#### 캘린더

| 섹션 | 빈 상태 텍스트 | CTA |
|---|---|---|
| 날짜별 일정 리스트 | "일정이 없습니다" | "+" 버튼 (일정 추가 바텀시트) |
| 일간 타임라인 | 타임라인만 표시, 일정 블록 없음 | 타임라인 빈 영역 탭 시 일정 추가 |

#### 투두

| 섹션 | 빈 상태 텍스트 | CTA |
|---|---|---|
| 할 일 목록 | "오늘 일정이 없습니다" / "새로운 일정을 추가해보세요!" | "+" 버튼 |
| 하루 일정표 | 완료율 0%, 오늘 일정 0으로 표시 | "+" 버튼 |

#### 습관

| 섹션 | 빈 상태 텍스트 | CTA |
|---|---|---|
| 오늘의 습관 | "아직 등록된 습관이 없어요" / "인기 습관으로 쉽게 시작해보세요!" | "인기 습관으로 시작하기" + "직접 만들기" |
| 습관 캘린더 | "습관을 등록하면 달성 기록이 여기에 표시돼요!" | 없음 (습관 등록 후 자동 표시) |

#### 루틴

| 섹션 | 빈 상태 텍스트 | CTA |
|---|---|---|
| 루틴 목록 | "아직 등록된 루틴이 없어요" / "반복되는 일정을 루틴으로 등록하면, 캘린더에 자동으로 표시돼요!" | "첫 루틴 만들기" |

#### 목표

| 섹션 | 빈 상태 텍스트 | CTA |
|---|---|---|
| 년간 목표 | "아직 등록된 목표가 없어요" / "올해의 목표를 세워보세요!" | "목표 추가하기" + "템플릿으로 시작하기" (v1.1) |
| 월간 목표 | "이번 달 목표가 없어요" / "년간 목표의 하위 목표를 설정해보세요!" | "하위 목표 추가하기" |
| 만다라트 | "만다라트로 목표를 체계적으로 구조화해보세요!" + 만다라트 설명 툴팁 | "만다라트 만들기" |

### 6.3 빈 상태 시각 톤

- 아이콘/일러스트는 앱의 메인 컬러 계열의 밝은 톤으로 표현한다.
- 텍스트는 중간 회색 톤으로, 과하게 눈에 띄지 않되 읽을 수 있는 수준으로 한다.
- CTA 버튼은 메인 컬러로 강조하여 다음 행동을 명확히 유도한다.
- 전체적으로 "비어있어서 쓸쓸하다"가 아니라 "시작하면 채워진다"는 긍정적 톤을 유지한다.

## 7. 에러 상태 처리 전략

### 7.1 에러 유형 분류

| 유형 | 원인 | 심각도 | 사용자 영향 |
|---|---|---|---|
| 네트워크 에러 | 인터넷 연결 끊김 | 높음 | 데이터 저장/불러오기 불가 |
| 인증 에러 | Google 토큰 만료, 세션 종료 | 높음 | 앱 사용 불가 |
| 동기화 에러 | Firestore 충돌, 오프라인 데이터 불일치 | 중간 | 최신 데이터 미반영 |
| 입력 에러 | 필수값 미입력, 형식 오류 | 낮음 | 해당 동작 실패 |
| 서버 에러 | Firebase 서비스 장애 | 높음 | 전체 기능 마비 |

### 7.2 에러별 사용자 대면 메시지 및 처리

#### 네트워크 에러

**표시 위치**: 화면 상단 스낵바 (지속 표시)

**메시지**: "인터넷 연결이 불안정해요. 연결을 확인해주세요."

**처리 전략**:
- Hive 로컬 캐시를 활용하여 읽기 동작은 오프라인에서도 가능하게 한다.
- 쓰기 동작(체크, 추가, 수정)은 로컬에 임시 저장하고, 네트워크 복구 시 자동 동기화한다.
- 상단에 "오프라인 모드" 표시 배너를 유지하여 사용자에게 현재 상태를 알린다.
- 네트워크 복구 시 "다시 연결되었어요!" 메시지를 잠시 표시 후 자동으로 사라지게 한다.

#### 인증 에러

**표시 위치**: 풀스크린 오버레이

**메시지**: "로그인이 만료되었어요. 다시 로그인해주세요."

**처리 전략**:
- Firebase Auth의 토큰 자동 갱신이 실패한 경우에만 발생한다.
- "다시 로그인" 버튼을 표시하고, 탭 시 Google 로그인 흐름을 재시작한다.
- 로그인 성공 시 이전 화면으로 복귀한다 (데이터 유실 없음).
- 로컬 캐시는 유지하여 재로그인 후 즉시 이전 데이터가 표시되게 한다.

#### 동기화 에러

**표시 위치**: 화면 하단 스낵바 (일시적)

**메시지**: "일부 데이터를 동기화하지 못했어요. 잠시 후 다시 시도할게요."

**처리 전략**:
- 자동 재시도 (30초 간격, 최대 3회).
- 3회 실패 시 "수동 동기화" 버튼을 표시한다.
- 충돌 발생 시 "마지막 수정 시간" 기준으로 최신 데이터를 우선한다 (last-write-wins).
- 심각한 데이터 충돌은 발생 가능성이 낮으나 (개인 사용 앱), 발생 시 서버 데이터를 우선한다.

#### 입력 에러

**표시 위치**: 해당 입력 필드 하단 인라인 에러

**메시지 예시**:
- 할 일 제목 미입력: "제목을 입력해주세요"
- 이름 미입력: "이름을 입력해주세요"
- 이름 20자 초과: "이름은 20자 이내로 입력해주세요"
- 시작 시간 > 종료 시간: "종료 시간이 시작 시간보다 이전이에요"
- 범위 일정 시작일 > 종료일: "종료일이 시작일보다 이전이에요"

**처리 전략**:
- 실시간 유효성 검사 (입력 중 표시, 포커스 아웃 시 검증).
- 에러 상태에서는 "저장" 버튼 비활성화.
- 에러 텍스트는 빨간색 계열로 표시.

#### 서버 에러 (Firebase 장애)

**표시 위치**: 풀스크린 에러 화면

**메시지**: "서비스에 일시적인 문제가 발생했어요. 잠시 후 다시 시도해주세요."

**처리 전략**:
- "다시 시도" 버튼을 제공한다.
- 로컬 캐시(Hive)가 있다면 읽기 전용 모드로 전환하여 기존 데이터는 열람 가능하게 한다.
- Firebase Status 페이지 링크를 제공하여 장애 상태를 확인할 수 있게 한다 (고급 사용자용).

### 7.3 에러 처리 UX 원칙

1. **사용자 탓으로 느껴지지 않게**: "잘못된 입력입니다" 대신 "OOO을 입력해주세요" 형태.
2. **다음 행동을 명확히 제시**: 에러 메시지와 함께 항상 "무엇을 하면 되는지" 안내.
3. **데이터 유실 방지 최우선**: 입력 중 네트워크 에러가 발생해도 작성 중인 내용은 로컬에 보존.
4. **에러 복구 시 부드러운 전환**: 에러가 해결되면 별도 조작 없이 자동으로 정상 상태 복귀.
5. **과도한 에러 표시 금지**: 사용자가 인지할 필요 없는 기술적 에러 (예: 캐시 미스)는 내부 로깅만 수행.

## 8. spec-architect / spec-security 의견에 대한 피드백

### 8.1 기술 아키텍처 관련 (spec-architect 방향 예상)

**상태 관리 (Riverpod)**:
- 사용자 관점에서 Riverpod 자체는 보이지 않는다. 중요한 것은 "화면 전환 시 데이터가 즉시 반영되는가"이다. 습관 체크 시 홈 대시보드의 도넛차트가 지연 없이 갱신되어야 한다. 이 부분의 반응성이 사용자 경험의 핵심이다.

**Firestore 구조**:
- 플랜 문서의 컬렉션 구조는 합리적이나, `habitLogs`가 날짜별로 무한히 쌓이는 구조다. 1년 사용 시 365개 문서가 생성된다. 성능에 문제가 없다면 괜찮으나, 습관 캘린더 화면에서 월간 데이터를 불러올 때 쿼리 최적화를 반드시 확인해야 한다.

**Hive 로컬 캐시**:
- 오프라인 지원은 사용자 가치가 매우 높다. 지하철 같은 불안정한 네트워크 환경에서 습관 체크가 안 되면 스트릭이 끊길 수 있다. 이 한 가지 상황만으로도 Hive 도입의 정당성은 충분하다.

### 8.2 보안 관련 (spec-security 방향 예상)

**Google OAuth Only**:
- 사용자 관점에서 Google 로그인만 제공하는 것은 "간편하다"는 장점과 "Google 계정이 없으면 못 쓴다"는 단점이 공존한다. 타겟층(20~30대 한국 사용자)의 Google 계정 보유율을 고려하면 현실적으로 문제없다. Apple 로그인은 iOS 출시 시 추가하면 된다.

**Firestore Rules**:
- 사용자 관점에서 보안 규칙이 중요한 이유는 "내 데이터가 타인에게 노출되지 않는다"는 신뢰이다. 개인 생산성 데이터(목표, 습관, 일정)는 매우 민감한 정보이므로, `users/{userId}` 하위 문서에 대한 소유자 전용 접근 규칙은 반드시 적용해야 한다.

**데이터 백업**:
- 사용자가 가장 두려워하는 것은 "데이터 유실"이다. Firestore 자체가 내구성이 높지만, 사용자에게 "데이터는 Google Cloud에 안전하게 보관됩니다" 정도의 안심 문구를 설정 화면에 표시하는 것을 권장한다.

## 9. 종합 제품 판단

### 9.1 이 제품이 성공하려면

1. **일일 사용 루프가 완성되어야 한다**: 아침에 열고 -> 하루 동안 체크하고 -> 저녁에 확인하는 루프. 이 루프가 한 곳이라도 끊기면 사용자는 이탈한다.

2. **빈 상태가 곧 온보딩이어야 한다**: 별도 튜토리얼은 이탈을 유발한다. 빈 상태 UI가 "다음에 무엇을 하면 되는지" 자연스럽게 안내해야 한다.

3. **스트릭이 DAU를 만든다**: 습관의 스트릭 카운터는 이 앱의 가장 강력한 리텐션 무기이다. "12일 연속을 끊고 싶지 않아서" 매일 앱을 여는 사용자가 코어 유저가 된다.

4. **크로스플랫폼 동기화가 매끄러워야 한다**: 웹에서 계획하고 앱에서 실행하는 사용 패턴이 자연스럽게 동작해야 한다. 동기화 지연이 느껴지면 신뢰를 잃는다.

5. **과도한 기능은 오히려 독이다**: 만다라트는 매력적이지만 복잡하다. MVP에서는 목표 리스트만으로 충분하다. "기능이 많다"가 아니라 "매일 쓰는 기능이 완벽하다"가 제품 가치이다.

### 9.2 경계해야 할 것

- **기능 과적재**: 5개 탭 x 각 2개 서브탭 = 10개 화면. 이미 상당한 규모이다. 추가 기능 요청에 대해 MVP 기준을 엄격히 지켜야 한다.
- **Notion 따라하기 함정**: 참고 이미지에 Notion 기반 템플릿이 다수 포함되어 있다. Notion의 자유도를 모바일 앱에서 재현하려 하면 실패한다. 구조화된 입력 폼이 모바일에서는 더 낫다.
- **디자인 화려함 vs 실용성**: Glassmorphism은 시각적으로 매력적이나, 도넛차트/타임라인/캘린더 그리드 등 정보 밀도가 높은 화면에서는 가독성을 해칠 수 있다. 정보 전달이 디자인보다 항상 우선이다.
